# 机票监控产品化打磨设计

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 把现有本地机票工具打磨成更安全、更适合长期运行的中文监控产品，支持持续定时查询、命中重复提醒、清爽仪表盘展示，以及更易理解的中文说明。

**Architecture:** 复用现有 FastAPI、SQLite、Playwright 和桌面通知链路，在不引入外部消息服务的前提下增强监控调度、命中去重、任务状态展示和首页体验。前端保持单页仪表盘结构，但将首页优先级调整为状态总览、最近命中、快速搜索和快速创建监控；后台则把监控调度从“能跑”提升到“可长时间稳定运行、可重复提醒、可观察”。

**Tech Stack:** FastAPI, Jinja2, 原生 JavaScript, SQLite, Playwright, `plyer`

---

## 目标

本次改造的目标不是新增一条完全新的产品线，而是把当前“能搜、能监控、能提醒”的本地工具打磨成一个普通用户更容易长期使用的机票监控工作台。重点是三件事：

1. 监控能持续运行，失败后不会卡死或停在旧时间点。
2. 命中后能重复提醒，但不会刷屏。
3. 首页和说明文档全部中文化，操作路径更清楚。

---

## 范围

### 包含
- 监控任务支持长期运行的状态更新与失败恢复
- 命中后按冷却时间重复提醒，而不是只在“第一次更低价”时提醒
- 桌面通知、页面高亮提示和声音提示
- 首页改为清爽仪表盘风格，强调登录状态、监控状态、最近命中和下一步操作
- 首页、导航、按钮、空状态、错误状态、说明文档全部中文化
- 监控任务列表、命中详情和搜索结果的中文表达与状态呈现
- 中文化发布说明和使用说明

### 不包含
- 外部通知渠道，例如邮件、微信、Telegram
- 自动购票或自动下单
- Windows 安装包或系统服务
- 多用户同步
- 迁移到前后端分离架构

---

## 关键决定

1. 监控仍然只针对本机常驻运行，不做云端任务队列。
2. 命中后允许重复提醒，但默认要有冷却时间，避免连续刷屏。
3. 监控任务失败后必须推进下次检查时间，避免卡在同一任务上反复报错。
4. 首页采用清爽仪表盘风格，而不是步骤向导或搜索优先布局。
5. 所有介绍性、说明性、状态性文案优先使用中文。
6. 不引入外部消息服务，保持部署和隐私边界简单。

## 监控与提醒设计

### 监控执行

监控调度器继续按任务的 `next_check_at` 扫描，但每次任务执行完成后必须更新：
- `last_checked_at`
- `next_check_at`
- `last_seen_lowest_price`
- `last_notified_at`
- `last_notified_price`

如果单个任务执行失败，也要更新 `last_checked_at` 和 `next_check_at`，确保任务在未来还能继续被检查，而不是卡死在失败点。

### 重复提醒

当前提醒策略改为“命中后按冷却时间重复提醒”：
- 首次命中时提醒
- 之后如果仍然命中，但距离上次提醒已超过冷却时间，则再次提醒
- 冷却时间建议默认 6 小时
- 如果价格进一步下降，则允许提前再次提醒

### 提醒内容

提醒必须包含：
- 路线
- 日期
- 当前最低价
- 目标价

提醒方式必须同时包含：
- 本机桌面通知
- 页面内高亮提示
- 声音提示

点击提醒后，优先打开本地页面中的对应监控任务详情或最近命中结果。

## 数据模型与存储

现有 `monitor_tasks` 和 `monitor_hits` 继续使用，但要补强状态字段的使用约定。

### `monitor_tasks`

已存在字段继续保留，重点使用：
- `enabled`
- `last_checked_at`
- `next_check_at`
- `last_seen_lowest_price`
- `last_notified_at`
- `last_notified_price`

### `monitor_hits`

继续保存命中快照，作为通知后回看依据。命中记录里至少保留：
- 命中时间
- 命中价格
- 最低价
- 航班快照

## 首页与前端设计

首页改为清爽仪表盘风格，首屏优先显示：

1. 登录状态
2. 监控状态
3. 最近命中
4. 快速搜索
5. 快速创建监控

### 首页状态文案

首页必须全部使用中文表达，避免下面这种英文式功能名：
- Search
- Results
- History
- Monitoring

要替换成：
- 搜索
- 结果
- 历史记录
- 监控任务

### 清爽仪表盘风格

视觉上要做这些调整：
- 降低页面卡片的厚重感
- 强化状态卡与主操作按钮
- 减少单一蓝色大面积堆叠
- 提高首屏信息层次
- 保持紧凑但不拥挤

### 页面可读性

所有标签、按钮、空状态、错误提示必须以中文优先展示。搜索与监控输入区仍可保留部分技术字段名，但面向用户的可见文本需要统一中文化。

---

## 交互与功能细节

### 搜索

保留现有搜索链路，但在界面上更偏“中文工作台”表达。结果区需要清楚展示：
- 当前最低价
- 航班数量
- 航班列表
- 跳转按钮

### 监控任务

监控任务列表应支持：
- 查看
- 编辑
- 启用 / 暂停
- 查看命中记录

监控详情页应突出：
- 当前任务条件
- 最近检查时间
- 最近最低价
- 最近命中结果
- 命中的航班列表

### 失败状态

失败提示要明确、可操作，不能只显示技术名词。比如：
- `携程登录已失效，请重新登录`
- `这次没有成功读取携程结果，请重试一次`
- `监控没有运行，请重新启动程序`

## 文档中文化

需要把仓库里的介绍性和说明性内容尽量改成中文，至少包括：
- `README.md`
- `README_使用说明.txt`
- 首页文案
- 空状态文案

保留必要的英文技术名词可以接受，但对普通用户可见的说明文字要中文优先。

必须覆盖以下行为：

1. 监控任务命中后会写入命中记录并触发提醒。

3. 任务失败后会推进下一次检查时间。

5. 前端继续避免 `innerHTML` 和不安全外链打开方式。
6. README 和使用说明改成中文后仍保留必要的启动与使用信息。

## 验收标准

完成后应满足：

- 用户可以长期运行监控任务，不会因为单次失败卡住。
- 当机票价格命中目标价时，会本机提醒并能回到网页继续查看。
- 命中提醒不会在短时间内无限重复，但在冷却后可以再次提醒。
- 首页一眼能看懂登录状态、监控状态和最近命中。
- 页面和 README 的说明文字大部分已经中文化。
- 现有测试和新增回归测试全部通过。
