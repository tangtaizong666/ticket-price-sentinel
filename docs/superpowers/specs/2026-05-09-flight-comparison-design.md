# 机票比价脚本与前端网站设计

## 1. 目标

构建一个本地可运行的机票比价工具，提供网页前端，支持用户输入国内单程固定日期的机票查询条件，自动抓取携程页面中的机票数据，筛选出符合条件的航班，展示当前最低价，并允许用户点击结果跳转到对应页面。

首版目标是让用户本人在本地稳定使用；如果效果满意，其他人也可以下载仓库后在自己的机器上本地运行。

## 2. 首版范围

### 包含
- 国内单程机票
- 固定出发日期
- 即时搜索比价
- 前端表单输入搜索条件
- 抓取携程搜索结果
- 按条件过滤结果
- 展示当前最低价
- 展示符合条件的航班列表
- 点击结果跳转到携程对应页面或对应搜索结果页
- 保存搜索历史
- 基于历史记录重新搜索
- 基于历史记录回填并修改条件
- 首次允许人工介入完成登录，之后尽量复用会话

### 不包含
- 往返票
- 多程票
- 国际、港澳台机票
- 定时监控与低价提醒
- 多用户账号体系
- 收藏航班
- 舱位、退改签、行李等深度筛选
- 线上多人部署方案

## 3. 关键产品决定

1. 第一版直接围绕真实携程抓取设计，不先做假数据演示版。
2. 数据获取方式采用浏览器自动化，工具为 Playwright。
3. 允许首次人工登录或验证，之后复用本地会话；不设计绕过验证码或风控的方案。
4. 历史记录保存的是搜索条件，不是航班结果快照。
5. 搜索结果以每次实时抓取到的数据为准。
6. 跳转优先落到具体航班页面；如果拿不到稳定深链，则退化到对应搜索结果页。

## 4. 总体架构

系统采用本地单体应用架构，由四个模块组成：

### 4.1 前端页面
负责接收用户输入、展示最低价与航班列表、展示和编辑历史记录。

### 4.2 后端 API
负责接收请求、校验输入、触发抓取、标准化与筛选航班数据、管理历史记录。

### 4.3 抓取模块
负责使用 Playwright 启动浏览器、复用登录态、执行携程搜索、提取航班数据与跳转信息。

### 4.4 本地存储
负责保存搜索历史与抓取会话状态，使用 SQLite。

## 5. 技术栈

- 后端：FastAPI
- 前端：Jinja2 模板 + 少量原生 JavaScript
- 抓取：Playwright
- 数据库：SQLite
- 运行方式：本地启动 Web 服务，通过浏览器访问

选择这套技术栈的原因：
- 抓取与后端都适合放在 Python 工程中实现
- 空项目起步成本低，交付速度快
- 前端不需要复杂构建流程
- 其他人下载仓库后更容易本地运行

## 6. 前端信息架构与交互

## 6.1 首页布局
首页采用三块结构：
- 左侧为搜索区
- 右侧为结果区
- 下方为历史搜索区

## 6.2 搜索区
搜索区分为两类输入：

### 基础条件
- 出发地
- 到达地
- 出发日期
- 最高价格

### 高级筛选
高级筛选采用“点击展开选项，选中后加入已选条件标签区”的交互方式。

包含：
- 起飞时间
- 航班属性（如直飞、经停）
- 航空公司

规则如下：
- 用户点击某个筛选块后展开可选项
- 选中的选项会显示在“已选条件”区域
- 未选中的选项不会出现在“已选条件”区域
- 用户可以在“已选条件”区域移除某个标签
- 提供“清空筛选”按钮，一次清除所有高级筛选条件

## 6.3 结果区
结果区展示：
- 当前最低价
- 符合条件的航班数量
- 航班结果列表

每条航班至少展示：
- 航班号
- 出发时间
- 到达时间
- 出发地与到达地
- 是否直飞或经停
- 航空公司
- 当前价格
- 跳转按钮

## 6.4 历史记录区
每条历史记录展示：
- 出发地
- 到达地
- 日期
- 价格上限
- 已选筛选标签
- 最近一次搜索时间

每条历史记录提供两个动作：
- 重新搜索：直接使用原条件再次抓取
- 编辑：将原条件回填到搜索表单，允许修改后再搜索

## 7. 搜索执行流程

1. 用户填写基础条件。
2. 用户按需展开并选择高级筛选条件。
3. 前端将搜索条件提交给后端。
4. 后端校验输入。
5. 后端触发抓取模块执行一次携程搜索。
6. 抓取模块返回原始航班数据。
7. 后端将原始数据标准化为统一结构。
8. 后端根据搜索条件执行过滤与排序。
9. 后端返回最低价、符合条件的列表和跳转信息。
10. 前端展示搜索结果。
11. 后端保存或更新本次搜索条件到历史记录。

## 8. 数据流与模块边界

### 前端与后端
前端只负责收集条件和展示结果，不直接处理抓取逻辑。

### 后端与抓取模块
后端通过明确接口调用抓取模块。抓取模块只负责执行页面操作和提取数据，不承担业务筛选逻辑。

### 后端与数据库
后端负责历史记录与抓取会话状态的读写。数据库不直接保存业务逻辑。

### 抓取模块与浏览器会话
抓取模块使用持久化浏览器用户目录保存登录态，避免每次重新登录。

## 9. 数据模型

## 9.1 搜索历史
建议字段：
- id
- origin_city
- destination_city
- departure_date
- max_price
- departure_time_filters
- flight_attribute_filters
- airline_filters
- last_searched_at
- created_at
- updated_at

说明：
- `departure_time_filters` 保存如上午、下午、晚上等标签集合
- `flight_attribute_filters` 保存如直飞、经停等标签集合
- `airline_filters` 保存用户选中的航空公司集合

## 9.2 抓取会话状态
建议字段：
- id
- browser_profile_path
- session_status
- last_successful_scrape_at
- updated_at

说明：
- `session_status` 用于标记当前登录态是否可用
- 不保存用户密码等敏感凭证

## 9.3 运行期航班结果
航班结果在运行期内存中处理，不作为长期持久化主数据。

统一结构至少包含：
- flight_no
- airline
- origin_city
- destination_city
- departure_time
- arrival_time
- is_direct
- stop_info
- price
- deeplink_url
- fallback_search_url

## 10. 后端接口设计

首版接口保持最小集合：

- `GET /`：渲染首页
- `POST /api/search`：执行搜索并返回结果
- `GET /api/history`：获取历史记录列表
- `POST /api/history/{id}/rerun`：使用历史条件重新搜索
- `GET /api/history/{id}`：获取单条历史记录详情，用于回填编辑
- `PUT /api/history/{id}`：更新历史记录中的搜索条件
- `POST /api/session/relogin`：触发重新登录流程

接口层规则：
- 所有输入都在后端再次校验
- 抓取失败返回明确错误状态，不以空列表冒充成功
- 无结果时返回成功状态和空结果列表

## 11. 抓取模块设计

抓取模块职责：
- 启动 Playwright 持久化浏览器上下文
- 复用本地登录态
- 在登录态失效时触发重新登录流程
- 打开携程搜索页
- 按输入条件完成搜索
- 提取页面中的航班数据
- 尽量提取具体航班跳转链接
- 无法提取稳定深链时生成对应搜索结果页跳转链接

实现约束：
- 第一版允许人工完成首次登录或再次登录
- 不实现验证码绕过、风控绕过或批量高频抓取能力
- 页面结构变化时，应显式报错并记录日志

## 12. 错误处理与失败场景

### 12.1 登录态失效
- 后端返回需要重新登录的错误状态
- 前端提示用户重新登录
- 用户可点击按钮触发浏览器打开并完成登录

### 12.2 抓取失败
- 页面结构变化、页面加载异常、提取失败等情况统一视为抓取失败
- 后端返回明确失败信息
- 前端保留当前条件，允许重试
- 记录抓取失败日志

### 12.3 无符合条件结果
- 返回成功状态
- 结果列表为空
- 页面显示无符合条件的提示
- 如果抓到了总体结果，可同时展示该次抓取到的最低价

### 12.4 输入不合法
- 前端执行基础校验
- 后端执行最终校验
- 错误信息贴近表单展示

## 13. 测试与验收

## 13.1 最低验证范围
- 用户可以输入条件并发起搜索
- 系统可以抓到航班列表并展示最低价
- 价格上限过滤有效
- 起飞时间过滤有效
- 直飞/经停过滤有效
- 航空公司过滤有效
- 点击航班可以跳转到对应页面或对应搜索结果页
- 搜索历史可以保存
- 历史记录可以重新搜索
- 历史记录可以回填并编辑
- 登录态失效时有明确提示
- 无结果时有明确空状态

## 13.2 验收标准
满足以下条件即可认为第一版完成：
- 用户能在网页中输入搜索条件
- 系统能通过浏览器自动化抓取携程国内单程固定日期机票
- 页面能展示符合条件的机票与当前最低价
- 用户点击结果能跳转到对应页面或对应搜索结果页
- 搜索记录能保留，并支持再次搜索和修改
- 抓取失败、未登录、无结果三类情况都有明确反馈

## 14. 后续扩展方向

首版完成后，可以按独立阶段继续扩展：
- 低价监控与定时任务
- 往返票支持
- 国际机票支持
- 结果缓存与搜索加速
- 多用户账号与私有在线部署
- 更丰富的筛选维度

## 15. 实施边界

本设计只覆盖第一版产品与技术方案，不包含：
- 具体页面样式精修
- 最终部署脚本
- 反爬绕过方案
- 第三方服务集成

第一版的成功标准不是覆盖所有机票场景，而是在可控范围内完成一个能本地使用的真实抓取比价闭环。
