# 机票监控、桌面提醒与一键启动设计

## 1. 目标

在现有本地机票比价项目基础上，新增一个本地可常驻运行的机票监控能力。用户可以保存监控任务，按自定义频率检查目标航线与日期的机票价格；当最低价低于或等于目标价时，程序在本机弹出桌面通知。用户点击通知后回到我们自己的网页查看本次命中的航班信息，再点击具体机票跳转到携程对应页面继续后续操作。

同时，为了让不懂代码的用户更容易使用，第一阶段还需要把项目交付形态提升到“下载后尽量少配置，双击即可启动”的水平。

## 2. 范围

### 包含
- 基于现有国内单程固定日期搜索链路新增监控功能
- 保存多个监控任务
- 用户自定义检查频率
- 后端后台常驻调度监控任务
- 监控命中后触发本机桌面通知
- 点击通知后打开本地网页中的监控任务详情或命中结果页
- 在网页中查看本次命中的航班列表与最低价
- 在网页中点击具体机票跳转到携程页面或对应搜索结果页
- 保存命中记录摘要，供通知后查看
- Windows 一键启动脚本（本地运行、自动打开浏览器）

### 不包含
- 邮件提醒
- 微信、Telegram、短信等外部通知
- 关机后继续监控
- 系统服务 / 开机自启动
- 正式 Windows 安装包或 exe 打包版
- 多用户同步
- 多平台通知适配优化
- 自动购票或自动下单

## 3. 关键产品决定

1. 第一版提醒方式为本机桌面通知，不接入外部消息渠道。
2. 第一版监控命中条件为：当前最低价小于或等于用户设定的目标价。
3. 检查频率由用户自定义。
4. 监控采用后端后台调度器，不依赖前端轮询。
5. 点击桌面通知后，不直接跳携程，而是先打开我们自己的网页查看命中结果。
6. 用户在网页中再点击具体机票，跳转到携程对应页面。
7. Windows 第一阶段交付形态为“双击启动脚本”，而不是安装包。
8. 后续如需扩展为独立 worker 架构，本阶段的模块边界需要预留拆分空间。

## 4. 总体架构

在现有系统上新增四个监控相关模块：

### 4.1 监控任务管理
负责创建、修改、启用、暂停、删除监控任务。

### 4.2 后台调度器
负责常驻运行、按任务的 `next_check_at` 扫描应执行的监控任务，并逐个触发检查。

### 4.3 监控执行器
负责复用现有抓取与筛选链路，执行一次真实搜索，计算当前最低价，并判断是否命中目标价。

### 4.4 桌面通知模块
负责在任务命中时发送本机桌面通知，并引导用户回到本地网页查看详情。

## 5. 数据流

核心流程如下：

1. 用户在网页中创建监控任务。
2. 后端将监控任务写入 SQLite。
3. 后台调度器定时检查哪些任务到达了 `next_check_at`。
4. 监控执行器调用现有抓取模块完成一次搜索。
5. 结果经过现有筛选逻辑后计算最低价。
6. 如果最低价小于或等于目标价，则写入一条命中记录。
7. 桌面通知模块发送本机提醒。
8. 用户点击通知后，浏览器打开本地网页中的任务详情页或命中结果页。
9. 用户在网页中查看航班列表并点击某条机票，跳转到携程对应页面。

## 6. 监控任务模型

建议新增 `monitor_tasks` 表，字段至少包含：

- `id`
- `origin_city`
- `destination_city`
- `departure_date`
- `target_price`
- `check_interval_minutes`
- `departure_time_filters`
- `flight_attribute_filters`
- `airline_filters`
- `enabled`
- `last_checked_at`
- `next_check_at`
- `last_seen_lowest_price`
- `last_notified_at`
- `last_notified_price`
- `created_at`
- `updated_at`

说明：
- 监控任务沿用现有搜索条件结构，只额外增加目标价、频率与运行状态字段。
- `enabled` 用于暂停与恢复任务。
- `next_check_at` 决定调度器下一次何时执行该任务。
- `last_notified_price` 用于提醒去重。

## 7. 命中记录模型

建议新增 `monitor_hits` 表，字段至少包含：

- `id`
- `monitor_task_id`
- `hit_price`
- `hit_at`
- `search_snapshot_json`
- `lowest_price`
- `created_at`

说明：
- `search_snapshot_json` 保存本次命中的航班摘要列表。
- 每条航班摘要至少保留：航班号、航司、时间、价格、是否直飞、跳转链接/回退搜索链接。
- 命中记录的主要用途是：点击通知后，不必重新抓取，也能先向用户展示“刚刚命中时”的结果。

## 8. 命中后的交互流程

1. 某监控任务命中目标价。
2. 程序弹出桌面通知，内容包括：
   - 路线
   - 日期
   - 当前最低价
   - 目标价
3. 用户点击通知。
4. 浏览器打开我们自己的本地网页，优先进入：
   - 该监控任务详情页，或
   - 最近一次命中结果页
5. 页面展示：
   - 当前命中时间
   - 当前最低价
   - 本次命中的航班列表
   - 原监控任务条件
6. 用户点击某条航班。
7. 打开携程对应页面继续查看或购买。

设计原则：通知只负责把用户“叫回来”，详细信息与操作承接放在我们的网页中完成。

## 9. 监控调度规则

### 9.1 常驻方式
- 程序主进程启动时，同时启动后台调度循环。
- 用户即使关闭网页标签，只要主进程仍在，本地监控继续运行。

### 9.2 调度规则
- 调度器按固定短周期扫描任务表。
- 每次只选择：
  - `enabled = true`
  - `next_check_at <= now`
  的任务。
- 对命中的任务逐个执行，避免同时把所有任务堆到一个时刻。

### 9.3 执行完成后更新
每次任务执行完成后更新：
- `last_checked_at`
- `next_check_at`
- `last_seen_lowest_price`

## 10. 提醒去重规则

为了避免重复刷屏，第一版建议仅在以下条件同时满足时触发提醒：

1. 当前最低价 `<= target_price`
2. 且满足以下任一条件：
   - 这是第一次命中
   - 当前最低价 `< last_notified_price`
   - 用户修改过任务条件并重新启用监控

这意味着：
- 首次 380 元命中会提醒
- 一直保持 380 元不会反复提醒
- 后续降到 360 元时再次提醒

### 11.2 监控任务详情页

## 12. 桌面通知设计

第一版优先按 Windows 本机桌面通知实现。

通知内容建议简洁，只包含：
- 标题：例如 `机票监控命中：北京 → 上海`
- 内容：例如 `当前最低价 ¥380，已达到你的目标价 ¥400`

点击通知后的行为：
- 打开本地网页
- 定位到对应的任务详情或命中结果页

## 13. 一键启动交付形态

第一阶段目标不是安装包，而是“普通用户下载后少配置、双击能运行”。

建议提供一个 Windows 启动脚本，例如：
- `start_fly_ticket.bat`

脚本职责：
1. 检查 Python 是否存在
2. 若无虚拟环境则自动创建
3. 安装依赖
4. 安装 Playwright Chromium
5. 若 `.env` 不存在则从 `.env.example` 自动生成
6. 启动本地 Web 服务
7. 启动后台监控调度器
8. 自动打开浏览器到本地页面

此阶段仍属于“开发者友好的本地工具”，但交付体验会明显优于命令行手动操作。

## 14. 与现有架构的关系

监控功能尽量复用现有链路：
- 搜索条件结构复用现有 `SearchRequest`
- 抓取复用现有 Playwright + Ctrip 解析链路
- 价格筛选复用现有最低价与过滤逻辑
- 会话管理复用现有登录态能力

新增的主要是：
- 监控任务表
- 命中记录表
- 后台调度器
- 通知发送逻辑
- 监控详情页与命中页

## 15. 测试与验收

### 15.1 最低验证范围
- 用户可以创建监控任务
- 用户可以选择频率、目标价并启用任务
- 后台调度器会按时间执行任务
- 当最低价小于等于目标价时，会写入命中记录
- 程序会触发桌面通知
- 点击通知可以打开本地网页
- 网页中可以看到本次命中的航班列表
- 点击具体机票可以跳转到携程对应页面或结果页
- 同价不重复提醒，降得更低时再次提醒

### 15.2 第一版验收标准
满足以下条件即可认为第一版完成：
- 本地程序运行时，可以后台持续监控
- 用户可保存多个监控任务
- 命中目标价时会触发本机桌面通知
- 点击通知能回到我们的网页查看命中结果
- 用户可在网页中点击具体航班跳到携程继续操作
- 项目提供 Windows 双击启动脚本，降低普通用户使用门槛

## 16. 后续扩展方向

本设计完成后，可继续按独立阶段扩展：
- 外部提醒渠道（邮件、微信、Telegram）
- 独立监控 worker 进程
- 开机自启动
- exe 打包版 / 安装包版
- 更多监控触发规则（价格下降提醒、组合条件提醒）
- 监控统计与价格趋势图

## 17. 实施边界

本设计聚焦：
- 本地后台监控
- 本机桌面提醒
- 点击通知回网页再跳携程
- Windows 一键启动

本设计不覆盖：
- 云端通知
- 多设备同步
- 自动购票
- 安装包级别交付

第一阶段的成功标准不是把它做成成熟商业软件，而是在现有本地抓取工具上，扩展出一个可实际使用的“本地价格监控与提醒闭环”。
